# 视频渠道定价兼容与生产发布设计

## 背景

`dev` 分支已经增加 Grok 视频模型测试和渠道视频定价界面，但合并前审查发现当前实现存在以下契约不一致：

- 前端提交 `billing_mode=video`，后端管理接口的请求校验仍拒绝该枚举值。
- 管理端和用户端把渠道视频价格解释为每秒价格，运行时渠道计费仍按请求次数计算。
- 模型广场转换分组数据时丢失视频倍率和分辨率价格字段。
- 用户端价格预览只读取分组覆盖价，没有按照运行时规则回退到渠道价格。
- 历史上使用 `billing_mode=image` 保存的视频模型会被前端自动改成 `video`，但原价格是按次价格，不能直接解释为每秒价格。
- 显式 `kind=image` 会截断基于视频模型名称的旧数据兜底，导致历史视频模型无法进入视频测试模式。

生产数据库当前存在一条 Grok 视频历史定价：`billing_mode=image`、`per_request_price=2.1`。该数值代表每次请求价格，不能自动换算或原值迁移为每秒价格。

## 目标

- `billing_mode=video` 明确表示视频每秒价格，并按照视频数量、规范化时长和有效倍率计费。
- 历史 `image/per_request` 视频配置继续按次计费，不改变现有扣费语义。
- 用户看到的价格来源和运行时实际扣费来源保持一致。
- 历史视频模型仍可出现在视频模式中，但不会在未确认价格单位的情况下自动改写持久化配置。
- 修复后的版本必须重新经过 staging 的接口、页面和真实计费验证，不能沿用旧 commit 的生产确认。

## 非目标

- 不自动修改生产或 staging 数据库中的历史渠道定价。
- 不根据固定时长把历史按次价格折算为每秒价格，因为历史请求时长不固定。
- 不修改分组视频价格字段的现有每秒语义。
- 不改变普通图片、普通按次或 token 模型的计费行为。

## 计费语义

### 新视频模式

渠道定价条目的 `billing_mode=video` 表示：

- `per_request_price` 是默认每秒价格。
- `intervals[].tier_label` 使用 `480p`、`720p`、`1080p`，对应字段 `per_request_price` 也是该分辨率的每秒价格。
- 运行时费用为：`每秒价格 * 视频数量 * 规范化时长秒数 * 有效倍率`。
- 分辨率层级未命中时回退到渠道默认每秒价格；渠道也未配置时回退到系统视频默认价。

保留现有字段名是为了复用渠道定价存储结构，API 文档和代码注释必须明确视频模式下字段单位为每秒，避免字段名导致误解。

### 历史模式

模型名包含视频但 `billing_mode=image` 或 `billing_mode=per_request` 的历史条目继续按请求次数计费：

- `per_request_price` 和分辨率层级价格仍为每次请求价格。
- 运行时不得乘视频时长。
- 管理端编辑时不得仅凭模型名称自动把模式改成 `video`。
- 管理员需要主动选择 `video` 并填写正确的每秒价格后，才能保存为新语义。

这保证生产现有 `2.1/次` 不会在 8 秒视频上被误计为 `16.8`。

## 管理端契约

- 管理接口请求 DTO 接受 `token`、`per_request`、`image`、`video`。
- 服务层把 `video` 纳入必填价格校验：必须存在默认价格或至少一个价格层级。
- `video` 继续复用当前非负价格和层级完整性校验。
- 创建、更新以及账号统计定价规则使用同一套枚举和服务校验。
- 管理端加载历史条目时保留后端返回的原始 `billing_mode`，不做隐式单位迁移。
- 只有管理员明确选择 `video` 后，界面才展示每秒单位文案。

## 用户模型分类

模型分类遵循以下优先级：

1. 模型名称明确属于 Grok 视频模型时归类为 `video`，用于兼容历史 `kind=image` 数据。
2. 否则使用后端显式 `kind=video` 或 `kind=image`。
3. 最后使用定价模式和现有图片模型名称规则兜底。

分类只决定页面展示和测试端点，不改变后端持久化定价模式，也不改变运行时计费模式。

## 用户价格解析

模型广场和模型测试台使用同一套视频价格解析规则，避免各页面各自拼装：

1. 当前分组对应分辨率的 `video_price_*` 覆盖价。
2. 当前模型渠道定价中匹配分辨率的层级价格。
3. 当前模型渠道定价的默认价格。
4. 已知 Grok 视频模型的系统默认每秒价格。

每一步都乘以当前有效倍率：

- `video_rate_independent=true` 时使用 `video_rate_multiplier`。
- 否则使用用户专属分组倍率或分组通用倍率。

价格预览再乘以用户选择的视频时长；价格卡的分辨率价保持显示每秒单价。历史 `image/per_request` 条目如果作为兼容视频模型展示，必须明确按次计算，不能套用每秒公式。

模型广场构造分组桶时必须完整保留 `video_rate_independent`、`video_rate_multiplier` 和三个 `video_price_*` 字段，后续同 ID 数据合并不得丢失已有字段。

## 错误处理

- 管理端提交无价格的 `video` 定价时返回稳定的用户参数错误，不允许形成零价渠道规则。
- 视频价格解析遇到空值时按既定优先级回退，不使用 `0` 代替缺失价格。
- 未知视频模型且没有任何价格来源时显示无价格，不猜测单价。
- 运行时渠道视频计费失败时保留现有日志与系统默认价兜底，但日志必须能够定位模型、分辨率和计费模式。

## 测试设计

所有修复遵循先失败、后实现的 TDD 流程：

- 管理端 HTTP JSON 绑定测试：`billing_mode=video` 可通过，未知枚举仍返回 400。
- 服务层校验测试：视频模式缺少默认价和层级时失败，有任一价格时通过。
- 运行时计费测试：`video` 渠道价乘视频时长；历史 `image/per_request` 视频价不乘时长。
- 模型分类测试：模型名为 Grok 视频时，即使后端 `kind=image` 也归类为视频。
- 模型广场测试：分组视频字段不会在转换和合并中丢失。
- 共享价格解析测试：覆盖分组价、渠道分辨率价、渠道默认价、系统默认价以及倍率优先级。
- 页面回归测试：视频模型只出现在视频模式；图片和文本模式不受影响。

完成定向测试后运行后端全量 Go 测试、前端全量 Vitest、ESLint、类型检查和生产构建。

## staging 与生产发布门禁

修复完成后只推送到 `origin/dev`，重新构建 `sub2api:staging-<commit>` 并持久更新 staging 的 `SUB2API_IMAGE`。

staging 至少验证：

- 镜像版本 commit 与 `origin/dev` 一致。
- 管理端可创建和更新 `billing_mode=video` 的渠道定价。
- 模型广场和模型测试台显示同一价格来源与单位。
- 使用测试 API Key 生成已知时长的视频后，用量日志费用等于预览公式且只扣费一次。
- 历史 `image/per_request` 视频条目仍按次计费，编辑页面不会自动改成 `video`。
- 健康接口、关键页面、管理接口、容器日志和数据库迁移状态正常。

只有修复后的 staging 验证完成并再次获得用户明确生产确认后，才允许合并到 `main` 并更新 prod。生产发布保留当前 `sub2api:prod-75b275528371` 作为回滚镜像，不执行历史价格数据迁移。

## 影响范围与风险

- 页面：管理端渠道定价、模型广场、模型测试台。
- 接口：管理端渠道创建/更新、用户可用渠道接口；不新增公开 API。
- 数据表：继续使用 `channel_model_pricing` 和 `channel_pricing_intervals`，不修改表结构、不批量更新数据。
- 运行时：只改变显式 `billing_mode=video` 的渠道计费；历史模式保持原语义。
- 主要风险：价格预览和真实扣费发生偏差、历史价格被误解释、分辨率回退不一致。通过共享解析规则、模式显式化和 staging 真实用量核对降低风险。
