# 分组图片 URL 响应设计

## 目标

在分组的图片设置中新增“默认传输方式”，支持 `b64_json` 和 `url`。历史分组及新建分组默认使用 `b64_json`；选择 `url` 时，网关将最终图片保存到服务端本地持久化目录，并向客户返回当前 API 域名下可公开访问、有效期为 24 小时的 URL。

## 范围

本功能覆盖 OpenAI 兼容图片端点：

- `POST /v1/images/generations`
- `POST /v1/images/edits`
- 上述端点的非流式与流式响应
- 原生 API Key、OAuth/Responses 适配和 Chat Completions 图片适配链路
- 图片超分与 2K/4K 二段增强后的最终响应

本功能不改变视频结果存储、批量图片任务存储、用户头像或管理端素材存储。

## 分组配置

分组新增字段 `image_response_format`，数据库与 Ent 默认值均为 `b64_json`，合法值仅为：

- `b64_json`：返回 `data[].b64_json`，保持当前兼容行为。
- `url`：将最终图片保存到本地并返回 `data[].url`。

旧数据库记录、旧 Redis API Key 鉴权缓存或未包含该字段的旧客户端请求出现空值时，统一按 `b64_json` 处理。管理 API 对非法值返回 400，不静默保存未知值。

管理端“分组 > 图片设置”新增“默认传输方式”选择项，默认显示 Base64，并说明客户显式传入 `response_format` 时以客户参数为准。

## 响应格式优先级

最终响应格式按以下顺序解析：

1. 客户请求显式传入的 `response_format`。
2. 当前 API Key 所属分组的 `image_response_format`。
3. 兼容默认值 `b64_json`。

客户显式参数只允许 `b64_json` 或 `url`。分组配置不得覆盖客户显式选择，避免管理员修改分组后破坏既有客户端的响应解析。

## URL 模式数据流

URL 模式必须统一处理上游可能返回的三种结果：裸 Base64、`data:image/*;base64,...` Data URL，以及 HTTP/HTTPS 图片 URL。

网关取得图片字节后，先完成当前分组配置的超分或 2K/4K 二段增强，再保存最终图片。不得在增强前保存基础图片并把旧 URL 返回给客户。

保存流程如下：

1. 校验单图大小，不超过现有图片下载上限 64MB。
2. 根据图片文件魔数识别真实格式，不信任上游文件名或 `Content-Type`。
3. 生成加密安全随机文件名；文件名不得包含用户、分组、模型、提示词或原文件名。
4. 在目标目录内写入临时文件，完成同步和权限设置后原子重命名。
5. 使用 API 端点地址的协议与主机生成公开 URL；若未配置 API 端点地址，则从当前请求经过可信代理处理后的协议和主机推导。
6. 返回 `https://<api-origin>/generated-images/<random>.<ext>`。

当本地目录不可写、磁盘不足、图片下载失败、格式无法识别或原子写入失败时，请求返回明确的系统错误，不自动降级为 Base64。日志记录时间、错误位置、请求和分组标识及错误堆栈，但不得记录图片内容或 Base64。

## 流式响应

`stream: true` 时，中间 `partial_image` 事件继续返回 Base64，不保存中间图片。最终 `completed` 事件只保存最终图片，并按 URL 模式返回本地公开 URL。

这样可以保留生成进度，同时避免为每个中间帧创建文件。最终图片保存失败时输出标准错误事件，不伪造成功完成事件，也不改为 Base64 完成事件。

## 本地存储

新增环境变量 `IMAGE_STORAGE_PATH` 指定生成图片目录。默认值位于应用数据目录下的 `generated-images`，不得写入源码目录、前端静态资源目录或容器临时构建目录。

staging 与 prod 必须挂载独立持久化目录，避免测试数据污染正式数据。容器重建后，未满 24 小时的链接应继续有效。

本方案支持单实例，或多个实例共同挂载同一存储目录。多个实例使用独立本地目录时，请求与后续图片访问可能落在不同实例并产生 404，因此不属于受支持部署方式。

## 公开访问端点

新增无需 API Key 的只读端点：

```text
GET /generated-images/:filename
```

访问端点只接受单层、符合随机文件命名规则且扩展名在允许列表中的文件名。路径解析必须阻止 `..`、绝对路径、编码绕过和符号链接逃逸。

文件链接属于不可枚举的 bearer URL。端点不提供目录列表、文件搜索、上传、删除或元数据接口。

响应使用图片真实 MIME 类型，并设置不超过剩余有效期的 `Cache-Control: private, max-age=<seconds>`。不允许设置超出文件有效期的公共 CDN 缓存。

## 过期与清理

图片保存时间固定为 24 小时，不提供分组级或客户级自定义 TTL。

公开端点每次访问时检查创建时间；满 24 小时立即返回 404，即使后台清理尚未执行。读取文件不得刷新其创建或过期时间。

后台清理服务在应用启动时执行一次清理，之后按固定周期扫描并删除过期图片。服务必须支持随应用优雅停止；单个文件删除失败只记录日志，不阻塞其他文件清理。

## 安全与并发

- 目录权限使用最小权限，文件不得具有可执行权限。
- 文件写入采用临时文件加原子重命名，防止读取半成品。
- 随机文件名提供足够熵，避免顺序枚举。
- 文件类型以内容识别，拒绝 HTML、JSON 或其他非图片响应。
- 清理与读取并发时，文件被删除可返回 404，不能触发路径回退或读取其他文件。
- 图片保存失败不得把敏感上游 URL、认证头或文件内容写入客户端错误信息。

## 影响链路

分组字段需要贯穿以下层级：

- 数据库迁移与 Ent schema/生成代码
- service Group 模型、repository 创建/更新与实体映射
- 管理端分组创建/更新 DTO、校验和输出 DTO
- API Key 鉴权 Redis 缓存快照及旧缓存兼容
- 前端 Group/CreateGroupRequest/UpdateGroupRequest 类型
- 分组创建和编辑表单、回填、重置及国际化文案

图片 URL 存储需要贯穿：

- 图片请求最终格式解析
- 原生 API Key、OAuth/Responses 与 Chat Completions 图片响应适配
- 非流式、流式完成事件、超分与二段增强后的统一落盘
- 本地图片存储服务、公开读取 handler、定时清理服务与应用生命周期
- Docker staging/prod 持久卷和部署文档

## 测试标准

后端至少覆盖：

- 新建分组默认值、显式 URL、更新保持、非法值拒绝。
- 数据库实体和 API Key Redis 缓存往返，旧空值回退 Base64。
- 客户显式参数优先于分组默认值。
- 裸 Base64、Data URL、上游 HTTP URL 均保存为本地最终图片。
- 图片格式识别、64MB 限制、随机命名、原子写入和存储失败错误。
- 公开读取成功、非法文件名、目录逃逸、非图片文件、过期文件和不存在文件。
- 启动清理、周期清理及删除失败不影响后续文件。
- 流式中间事件保持 Base64，最终事件返回 URL。
- 超分和 2K/4K 增强后保存最终图片，而不是基础图片。

前端至少覆盖：

- 创建分组默认选择 Base64。
- 编辑 URL 分组时正确回填。
- 创建和更新请求正确提交字段。
- 页面显示客户显式参数优先的说明。

完成后运行相关 Go 单元测试、前端单元测试、前端类型检查和项目构建；同时更新图片网关、可用渠道与源码部署文档。

## 验收标准

- 历史分组和未传 `response_format` 的历史客户保持 Base64 响应。
- 分组选择 URL 后，未显式传格式的客户收到当前 API 域名下的公开 URL。
- 客户显式选择 Base64 或 URL 时，其选择覆盖分组默认值。
- URL 在文件创建后的 24 小时内可直接访问，满 24 小时后返回 404。
- 流式请求只有最终图片落盘，中间图片不落盘。
- 存储失败返回明确错误，不返回错误格式或半成品 URL。
- staging 与 prod 图片目录隔离，容器重建不提前破坏有效链接。
