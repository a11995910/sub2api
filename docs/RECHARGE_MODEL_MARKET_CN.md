# 灵石余额、兑换码、订阅兑换与模型广场说明

## 功能范围

用户侧核心入口：

- `兑换码兑换`：用户在 `/redeem` 页面兑换余额兑换码或订阅兑换码。
- `在线充值`：用户在 `/payment` 页面使用内置支付渠道直接充值灵石余额或购买订阅，入口受 `payment_enabled` 控制。
- `我的订阅`：用户在 `/subscriptions` 页面查看订阅用量，并直接使用灵石兑换可售订阅套餐。
- `模型广场`：用户在 `/models` 查看当前可调用模型、可用分组，以及按用户有效倍率计算后的灵石价格。
- `模型测试台`：用户在 `/model-test` 选择模型、分组和对应 API Key，通过真实 `/v1` 网关在线测试文本、图片或视频模型。

用户侧菜单已不再展示“灵石充值”、“可用渠道”和“渠道状态”入口，相关后端接口、调度逻辑和管理员能力不受影响。`/purchase` 是历史兼容路由，访问时跳转到 `/redeem`，不再加载外部小铺 iframe。

## 充值与兑换入口

兑换页文件：`frontend/src/views/user/RedeemView.vue`

用户充值和兑换入口按渠道区分：

- `/redeem`：用户输入兑换码后兑换灵石余额或订阅。
- `/payment`：内置支付渠道可用时，用户直接在线充值灵石余额或购买订阅。余额充值的快捷金额只展示到 500；用户自定义输入超过 500 时，前端不创建自助支付订单，并提示单次支付最高 500 元、可多次 500 元支付。
- `/subscriptions`：用户查看订阅用量，并使用灵石兑换可售订阅套餐。
- `/purchase`：历史兼容地址，前端路由直接跳转到 `/redeem`。

订阅套餐统一在“我的订阅”页面用灵石兑换；内置支付的余额充值和订阅购买入口独立放在 `/payment`，避免和兑换码入口互相覆盖。

管理端“系统设置 → 支付设置”维护支付金额相关配置：

- `payment_balance_recharge_multiplier`：余额充值倍率，表示用户每支付 1 CNY 可获得多少灵石余额，只影响余额充值到账，不影响订阅套餐价格。
- `payment_subscription_usd_to_cny_rate`：订阅 CNY 换算汇率，保存到支付配置 `SUBSCRIPTION_USD_TO_CNY_RATE`。值为 `0` 或未配置时，订阅订单按套餐 `price` 数值直接收款；值大于 `0` 且支付实例币种为 CNY 时，订阅订单的网关扣款基数按 `套餐 price × 汇率` 计算。
- `payment_recharge_fee_rate`：充值手续费率，保存到支付配置 `RECHARGE_FEE_RATE`。余额充值和订阅购买都会在网关扣款基数上追加手续费；手续费按支付币种精度向上取整到最小货币单位。

管理端订阅套餐编辑弹窗会根据 `subscription_usd_to_cny_rate` 和 `recharge_fee_rate` 展示 CNY 通道实扣预览。预览只用于帮助管理员核对用户实际付款金额，不改变套餐 `price`、灵石余额扣减、订阅额度、兑换码面额或后端套餐保存结构。

## 余额兑换码与邀请返利

用户在 `/redeem` 使用 `balance` 类型且面额大于 0 的兑换码时，系统会先在数据库事务中标记兑换码已使用并增加用户灵石余额。兑换成功后，系统按当前邀请返利策略为该用户绑定的上游邀请人累计返利额度，并在返利流水中记录来源兑换码。

兑换码返利遵循邀请返利的统一规则：

- `affiliate_enabled` 关闭时不产生新返利。
- 用户未绑定上游邀请人时不产生返利。
- 返利金额按邀请人的专属返利比例优先计算；未设置专属比例时使用全局返利比例。
- 配置返利有效期时，超过被邀请用户注册后的有效期不再产生返利。
- 配置单个被邀请人返利上限时，系统会按剩余额度截断返利金额。
- 配置冻结期时，返利先进入冻结额度，到期后再解冻为可提额度。

支付订单充值仍由订单履约流程发放邀请返利，并使用订单来源记录返利流水。订单履约中自动使用的充值兑换码会跳过兑换层返利，避免同一笔支付重复返利。

如果管理端开启邀请返利并配置了“充值奖励分组”和“充值奖励天数”，被邀请用户首次通过内置支付订单或正数余额兑换码产生返利时，系统会为邀请人发放一次对应分组的限时使用权；同一邀请人和同一被邀请人只领取一次，后续充值或兑换仍可按返利规则继续产生金额返利，但不会再次累加奖励分组时间。奖励分组可以是标准专属分组或订阅分组：标准专属分组只授予访问权限，请求仍按该分组倍率正常消耗邀请人的余额；订阅分组则复用订阅续期规则，按订阅额度消耗。邀请人已有同类未过期限时授权时，首次奖励会从当前过期时间继续累加天数，已有但已过期时从当前时间重新计算；如果管理员已经永久授予该专属分组，则奖励不会覆盖永久授权。用户侧 `/affiliate` 邀请返利页面会用醒目的奖励卡片展示当前可获得的充值奖励分组、奖励天数和消耗方式；当邀请人已经拥有该奖励的限时授权或订阅时，卡片会显示当前剩余倒计时。用户侧 `/groups/available` 会返回 `access_expires_at` 和 `access_source`，API Key 创建/编辑和行内切换分组下拉会据此显示限时授权剩余时间；公开分组、永久授权和未携带到期时间的分组不显示倒计时。订单履约会写入 `AFFILIATE_GROUP_ACCESS_REWARD_APPLIED` 或 `AFFILIATE_GROUP_ACCESS_REWARD_SKIPPED` 审计记录，保证同一支付订单重试时不会重复累加天数；支付订单和兑换码共用 `AFFILIATE_GROUP_ACCESS_REWARD_CLAIMED` 业务幂等记录，保证同一被邀请人跨渠道后续充值不再重复发放奖励分组。兑换码返利发放分组奖励属于兑换成功后尽力执行的后置流程，兑换本身不会因为分组奖励失败而回滚；奖励发放失败时不会写入领取记录，后续可随新的有效返利再次尝试。标准专属分组奖励到期后，后台任务会把该用户仍绑定在奖励分组上的 API Key 迁移到 `API Key 默认分组`，并移除这条限时授权；未配置有效默认分组时会暂缓过期清理，避免 Key 失去可用分组。

管理端也可以直接查看和维护标准专属分组的授权有效期：

- 用户管理的“专属分组/倍率”配置会展示用户当前已授权专属分组的来源和剩余时间；管理员可以把授权设置为永久，或设置一个未来到期时间。
- 用户保存接口 `PUT /api/v1/admin/users/:id` 支持 `allowed_group_access` 字段。提交该字段时，后端会用其中的 `group_id` 同步用户专属分组集合；`expires_at: null` 表示永久授权，未来时间表示限时授权。已过期或无效时间会被拒绝，避免写入不可用授权。
- 分组管理的标准专属分组行提供“用户”入口，调用 `GET /api/v1/admin/groups/:id/allowed-users` 查看当前仍有效的授权用户、授权来源、剩余时间、到期时间和更新时间；已过期授权不会出现在该列表中。

`PUT /api/v1/admin/users/:id` 的专属分组授权示例：

```json
{
  "allowed_groups": [12, 15],
  "allowed_group_access": [
    {
      "group_id": 12,
      "expires_at": null,
      "source": "manual",
      "notes": ""
    },
    {
      "group_id": 15,
      "expires_at": "2026-07-12T10:00:00Z",
      "source": "manual",
      "notes": "限时试用"
    }
  ]
}
```

兼容旧前端或旧脚本只提交 `allowed_groups` 的场景：旧字段仍只负责同步授权分组 ID 集合，不会覆盖已存在授权的 `expires_at`、`source` 和备注等元数据，避免把邀请奖励产生的限时授权误改为永久授权。

管理端邀请返利的“返利记录”展示两类来源：

- `订单`：来源于内置支付订单充值，显示订单号、外部交易号、支付方式和订单状态。
- `兑换码`：来源于用户站内兑换余额兑换码，显示兑换码 ID、兑换码内容、兑换金额和兑换状态。

## 订阅兑换

订阅页面文件：`frontend/src/views/user/SubscriptionsView.vue`

后端接口：

```text
POST /api/v1/payment/plans/:id/purchase-with-balance
```

兑换流程：

- 前端读取 `/api/v1/payment/plans` 展示所有可售订阅套餐，保留套餐原价格、划线原价、描述、有效期、特性和关联分组信息。
- 用户点击“用灵石兑换”后会先看到确认弹窗；用户确认后，前端才调用兑换接口。
- 后端校验套餐仍可售、分组仍为启用的订阅分组、用户状态正常且灵石余额充足。
- 后端在同一数据库事务中扣减用户灵石、生成一张 `subscription` 类型兑换码、标记兑换码已使用，并创建或续期对应用户订阅。
- 兑换码面额记录为本次套餐价格，备注会写入套餐名称和套餐 ID，管理员后续可以在兑换码记录中查看这笔订阅兑换。
- 用户余额不足时不会创建兑换码，也不会修改订阅。

用户通过 `/payment` 在线购买订阅套餐时，后端会先校验套餐仍可售、关联分组有效且支付通道可用。订单 `amount` 记录套餐 `price`，`pay_amount` 记录网关实际收款金额：未配置订阅 CNY 换算汇率时按 `price` 直付；配置汇率且最终选中的支付实例币种为 CNY 时按 `price × subscription_usd_to_cny_rate` 作为扣款基数，再叠加 `recharge_fee_rate` 手续费。非 CNY 支付实例不执行订阅汇率换算。

## 模型广场

模型广场文件：`frontend/src/views/user/ModelMarketView.vue`

页面复用现有用户接口，不新增后端接口：

- `/api/v1/channels/available`：读取用户可访问分组与支持模型定价。
- `/api/v1/groups/available`：读取当前用户可绑定的完整可用分组，作为顶部“分组优先”导航的数据源。
- `/api/v1/groups/rates`：读取用户专属分组倍率。

页面展示改为“分组优先”：

- 顶部横向展示当前用户可访问的分组，支持按分组名、平台名和模型名搜索。
- 顶部分组以 `/groups/available` 为准展示完整用户可用分组；若某个分组暂时没有匹配到渠道模型，选中后展示空状态。
- 图片生成分组排在普通文本分组之后，用户选中一个分组后，下方只展示该分组可调用的模型卡片。
- 模型卡片展示平台、计费模式、阶梯数量、当前分组倍率、倍率后的灵石价格、官方美元价格和优惠幅度。
- 同一个分组内的模型按普通文本/按次模型优先、图片模型最后展示，便于用户先查看常用文本模型。
- 页面不再展示“最优价格”或跨分组价格对比，避免在新增分组渠道后误导用户。

价格展示规则：

- 基础定价来源于渠道模型定价配置。
- 文本、Token、按次价格使用用户有效文本倍率：优先用户专属倍率，其次分组默认倍率。
- 图片价格在分组启用独立图片倍率时使用图片倍率，否则使用文本倍率。
- 视频价格在分组启用独立视频倍率时使用视频倍率，否则使用用户有效文本倍率。视频输出报价依次读取当前分组的分辨率覆盖价、渠道分辨率层级价、渠道默认价和已知 Grok 模型系统默认价；参考图可以继续用于图生视频，但不额外收费。
- 官方价格按同一基础定价直接以美元 `$` 展示；优惠幅度按 `当前灵石价格 / (官方美元价格 * 7.1)` 折算，低于官方折算价时展示红色优惠比例。
- 图片生成能力作为独立模型展示：OpenAI 图片分组会生成 `image-2` 条目，普通 GPT 模型只展示非图片分组，避免把 `ChatGPT2API 图片` 这类图片分组挂到所有 token 模型下。
- 用户余额、实际扣费、充值、订阅兑换和模型价格展示为灵石；用量仪表盘和使用记录中的原价/标准成本用于对照上游原始成本，展示为美元。

图片模型卡片会直接按当前分组展示 `1K / 2K / 4K` 三档图片价格；未配置的档位展示 `-`，真实扣费仍按后端图片计费回退规则处理。每个模型卡片提供“去测试”按钮，会跳转到模型测试台并预选对应模型与分组。

视频模型卡片按官方能力展示输出分辨率：`grok-imagine-video` 展示 `480p / 720p`，`grok-imagine-video-1.5` 展示 `480p / 720p / 1080p`，不展示参考图价格。分组分辨率覆盖价、显式 `billing_mode=video` 的渠道层级价和渠道默认价均按秒展示；历史 `billing_mode=image/per_request` 的渠道价按次展示，不会套用按秒总价公式。`billing_mode=token` 不作为视频价格来源；历史按次模式缺少当前分辨率层级价和默认价时也不回退系统每秒价，避免把不同单位拼接为同一报价。

模型广场只展示模型、平台、分组、价格和测试入口，不展示渠道名称、渠道状态或上游账号状态，避免把调度细节暴露给普通用户。

## 用户侧自定义菜单

管理员可在系统设置中维护自定义菜单项。用户可见菜单按 `custom_menu_items.sort_order` 升序排列，并在侧边栏中显示于“模型广场”之后、“模型测试台”之前；管理员可见菜单同样按 `sort_order` 升序排列，并显示于管理端“系统设置”之前。菜单点击后进入 `/custom/{id}`，页面按配置的外部 URL 或 `md:{slug}` Markdown 页面加载内容。

## 模型测试台

模型测试台文件：`frontend/src/views/user/ModelTestView.vue`

页面复用现有用户接口和 OpenAI 兼容网关：

- `/api/v1/channels/available`：读取用户可见模型、可用分组及图片、视频价格。
- `/api/v1/groups/rates`：读取用户专属分组倍率。
- `/api/v1/keys`：读取当前用户 active API Key，作为测试入口的主选择项。
- `/v1/chat/completions`：文本模型测试端点。
- `/v1/images/generations`：图片模型测试端点。
- `/v1/images/edits`：上传参考图后的图片编辑测试端点。
- `/v1/videos`：通用视频生成任务创建端点；`/v1/videos/generations` 保留为兼容别名。
- `/v1/videos/{task_id}`：视频任务状态查询端点。
- `/v1/videos/{task_id}/content`：使用任务绑定的原创建账号代理读取已生成的视频内容。

测试台以用户已有 active API Key 为主入口。用户先选择 Key，页面自动带出该 Key 绑定的分组，并只展示此分组下可用的模型供用户选择。测试请求使用用户选中的真实 API Key 发起，因此会自然经过 API Key 鉴权、分组路由、限流、用量记录和灵石扣费。若没有 active API Key，或从模型广场跳转到某个分组但该分组没有 active API Key，页面只提示用户去 API Key 页面创建或绑定 Key，不会用其他分组的 Key 代替，避免测试流量落到错误分组。

页面按接口意图区分文本、图片和视频模式。文本与图片模式继续按模型分类；用户主动选择视频模式后，当前 API Key 的 `/v1/models` 返回的全部模型都可以被选择，不要求模型名包含 `video`，也不要求前端预先知道 Seedance、Jing、Kling、Veo 或 Sora 等供应商名称。没有视频定价的未知模型显示 `-`，但不阻止真实测试；未知模型默认禁用参考图，直到能力规则明确支持。

图片测试支持固定尺寸和自适应尺寸。固定尺寸会向图片网关传递用户当前选择的 `size`；自适应尺寸不会传递 `size`，适用于用户在提示词中自行描述尺寸、画幅或比例的场景。页面不会因为切换分组或模型自动改写图片尺寸，后端图片增强、计费和记录均以用户实际提交的尺寸参数为准。自适应尺寸的价格预览按后端当前默认 `2K` 分档展示，真实请求仍经过网关的图片计费记录链路。

图片模式可上传参考图片。未上传参考图时调用 `/v1/images/generations` 生成新图；上传本地参考图后调用 `/v1/images/edits`，以 multipart 表单提交 `model`、`prompt`、可选 `size`、`n=1`、`response_format=b64_json` 和图片文件。远程参考图可通过 JSON `images[].image_url` 提交到网关；当上游账号为 OpenAI APIKey 类型时，网关会校验远程地址为公网 HTTP/HTTPS 图片地址，并由服务器直接下载图片内容后转成 multipart 文件上传给上游。远程地址不可访问、返回非 2xx、返回 HTML/鉴权页等非图片内容或超过大小限制时，网关返回 `400 invalid_request_error`，不会把本地输入校验失败显示为上游 `502`。开启 `features_config.openai_images_upstream.mode=chat_completions` 的渠道会把图片入口转发到上游 `/v1/chat/completions`：文生图请求发送文本消息，图生图请求发送文本加 `image_url` 多模态消息，保证参考图随请求传递给上游。前端限制最多 4 张参考图，单张不超过 20MB，与后端图片上传单文件限制保持一致。

视频模式支持选择 `1-15` 秒时长；已知计费模型继续按能力显示分辨率，未知模型使用默认选项。测试台只向 `/v1/videos` 创建一次任务，再按返回的 `task_id` 每 5 秒轮询 `/v1/videos/{task_id}`；任务完成后统一通过 `/v1/videos/{task_id}/content` 携带当前 API Key 获取视频 Blob 并播放。超时错误保留任务 ID，失败、取消或超时都不会重新提交创建请求。

OpenAI 平台账号会先经过现有模型映射，再优先调用上游 `/v1/videos`。协议结果按账号和映射后模型缓存 24 小时；只有明确 `404`、`405` 或 `unsupported_endpoint` 才回退一次旧 Chat Completions 视频协议。业务 `400`、鉴权或额度错误、上游故障和网络超时都不回退，避免重复生成。创建成功先按用户、API Key、分组和任务 ID 绑定原账号，再交付任务并记录一次视频用量；轮询与内容读取不重复扣费。状态响应隐藏上游签名 URL，内容端点优先调用上游 `/content`，缺失时只接受公网 HTTPS 结果 URL，并限制重定向、媒体类型和 `gateway.upstream_response_read_max_bytes` 配置的最大响应体，不提供任意 URL 代理能力。

测试台按视频模型能力提供两类图片入口：`grok-imagine-video-1.5` 及其版本别名使用 image-to-video 工作流，最多上传 1 张起始图，并以官方 `image.url` 结构提交；标准 `grok-imagine-video` 使用 reference-to-video 工作流，测试台最多上传 4 张参考图，并以 `reference_images[].url` 结构提交，图片用于人物、物品或风格参考，不固定为视频首帧；未知视频模型会禁用图片上传入口。旧客户端使用的 `image.image_url` 会由网关兼容转换。视频图片单张原文件不超过图片上传总限制 20MB；超过 1MB 的图片会在浏览器中提示并自动缩放、转换为 JPEG，压缩到 1MB 以内后再上传。压缩失败时不会提交生成请求。非测试台客户端提交超过 1MB 的内联 data URL 时，网关在账号调度和计费前返回 `413 invalid_request_error`，避免把上游请求体限制显示成 `502`。

标准版 `grok-imagine-video` 的官方参考图引导工作流与起始图不同：API 客户端使用 `reference_images[].url` 提交，图片只影响人物、物品或风格，不固定为视频首帧；`grok-imagine-video-1.5` 不接受 `reference_images`。网关拒绝同一请求同时携带 `image` 和 `reference_images`，模型与图片模式不匹配时返回 `400 invalid_request_error`。参考图不产生额外费用，视频实际扣费仅按服务端规范化后的分辨率、时长、视频数量和有效倍率计算；状态查询和视频内容读取不重复扣费。

渠道定价配置提供独立的 `video` 计费模式。该模式的默认价格和 `480p / 720p / 1080p` 层级价格均按美元每秒（USD/s）填写；`per_request_price` 字段名仅为复用现有存储结构。参考图不产生额外费用，管理端不再提供参考图价格设置；历史配置中的 `input_price` 字段仅为兼容旧数据，加载时清空，保存时提交为空值，不参与模型广场、测试台或服务端计费。历史上仅包含视频模型但保存为 `image` 或 `per_request` 的渠道配置继续按美元每请求（USD/request）计费，管理端编辑时保留原模式且不自动迁移。管理员需要确认价格单位并重新填写每秒价格后，再主动切换为 `video`。

视频报价的完整优先级为：当前分组对应分辨率覆盖价、渠道对应分辨率层级价、渠道默认价、已知 Grok 模型系统默认价。分组覆盖价按秒；渠道层级价和默认价的单位由渠道 `billing_mode` 决定。显式 `video` 缺少当前层级价和默认价时可以回退系统每秒价；历史 `image/per_request` 缺价时不跨单位回退；`token` 模式直接视为无视频报价。最终报价使用视频独立倍率，或按“用户专属分组倍率优先、分组通用倍率兜底”的规则计算。

`grok-imagine-video-1.5` 未上传起始图时，网关会按标准模型 `grok-imagine-video` 路由和计费，测试台预览也使用标准模型价格；上传起始图执行图生视频时才保留 `grok-imagine-video-1.5` 的路由和价格，但参考图不额外收费。模型广场独立展示 1.5 的完整图生视频输出价目。

## 货币单位

前端统一通过 `formatSpiritStones()` 或 `common.currencyName` 展示余额、用量、兑换码面额和后台金额字段。历史字段名如 `*_usd` 暂不变更，原因是它们已经参与接口、数据库字段、统计逻辑和旧数据兼容；本次仅调整用户可见单位文案。

## 影响范围

- 用户侧：菜单、在线充值页、兑换入口、模型广场、模型测试台、余额和用量展示。
- 管理侧：用户余额、分组额度、订阅额度、兑换码、邀请返利记录、推广码、订单、统计图表等金额展示文案。
- 后端：余额兑换码成功兑换后纳入邀请返利；支付订单返利继续走订单履约链路；返利流水保留订单来源和兑换码来源，便于后台审计；不改变渠道调度逻辑。
